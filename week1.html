Compiling Benchmark.f and feq-parse with LFortran

GSoC 2026 @Fortran-lang

by Jatin Agarwal

Week 1 - Resolving module deserialization bugs for benchmark.f

May 17 – May 23, 2026

Welcome to the first blog post of my GSoC 2026 journey! This summer, I am working with the Fortran-lang organization to compile the benchmark.f and feq-parse packages using the LFortran compiler. The project aims to enable full builds of these packages by resolving parsing, semantic, and compilation issues. This week, I focused on getting benchmark.f to compile by tackling a series of Internal Compiler Errors (ICEs) related to module deserialization and external symbols.

Work done in this week:

  • Fixed ~unlimited_polymorphic_type scope resolution: When a derived type with class(*), allocatable (unlimited polymorphic) was defined inside a module, LFortran created a synthetic ~unlimited_polymorphic_type struct inside the struct's scope. During .mod deserialization, the compiler couldn't resolve it because the module name was recorded incorrectly. To help isolate the problem, I extracted a minimal reproducer for this bug (Issue #11332). Initially, I added a workaround to let the deserializer fall back and search the current scope (PR #11333). However, after discussing with reviewers, we realized the struct was being declared at the wrong scope level. The root cause fix was to walk up the scope chain and create the struct at the module/program scope instead, reusing existing ones if available.
  • Corrected ExternalSymbol generation for struct members: ExternalSymbol instances for struct members were being created with the struct's name as their m_module_name, confusing the deserializer. After extracting a minimal reproducer for this bug (Issue #11430) and making an initial attempt to patch the deserializer with a fall-through hack, we implemented the proper fix in the symbol generation logic itself. We explicitly set m_module_name to the actual owning module's name and used scope_names to encode the path to the struct. This explicitly tells the deserializer how to resolve the nested structure (PR #11436).
  • Resolved procedure pointer cross-module deserialization ICE: The compiler was crashing when a procedure pointer inside a derived type was associated via the => operator. After extracting a minimal reproducer for this bug (Issue #11607), I traced the AST-to-ASR generation and found that the body visitor was unwrapping the ExternalSymbol, directly assigning the raw function from the foreign module. This leaked the foreign module's symbol table ID into our serialized .mod file, causing a crash upon loading. The fix was to preserve the ExternalSymbol wrapper during procedure pointer association, keeping the reference safely bounded within the current module's symbol table (PR #11611).

Progress and Next Steps

With these serialization bugs resolved, the compilation of benchmark.f jumped from 76% to 86%!

However, we are not at 100% yet. Moving past the deserialization phase, we hit a new semantic error: 

Semantic error: Arguments do not match for any generic procedure, run

This occurs when the generic procedure run is called with (100000000000_i8, compute_pi). Next week, my focus will shift from fixing serialization ICEs to debugging generic procedure overload resolution in the semantic analyzer.
