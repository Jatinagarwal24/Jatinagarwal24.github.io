Week 8 - Fixing Procedure Pointers, ASR Verify Errors, and Multiple Definition Linker Issues

July 4 – July 10, 2026

This week, we focused heavily on resolving complex ASR verification failures related to struct symbols, fixing LLVM procedure pointer bitcasting issues, and ironing out multiple definition errors occurring during the separate compilation of Fortran files.

Resolving ASR Verify Error for StructType in Implicit Interface Dummy Args (PR #11975)

The Problem: When generating dummy arguments for implicit procedure interfaces, variables that had a StructType (like class(*)) were being created without their underlying struct symbol (passing nullptr instead). This missing information caused ASR verification failures. My initial attempt tried to fetch a random struct symbol to satisfy the verifier, but since StructType doesn't hold the exact symbol information, that approach was flawed.

The Solution: I realized that guessing the struct symbol wasn't the right approach. Instead, I updated the code to explicitly pass the exact type_declaration symbol directly into the interface creation function from wherever it's called. If a symbol isn't available, I just pass nullptr. This ensures the interface always gets the correct type information. I also fixed a syncing issue by simply skipping the sync step whenever we deal with a StructType. These changes cleanly resolved the ASR verifier errors! I also took some time this week to address the reviewer comments on my previous PR #11805.

Fixing Procedure Pointer Verification Failure by Bitcasting Fixed-Size Arrays (PR #12110)

The Problem: I ran into an LLVM verification failure when passing fixed-size arrays to a procedure pointer: Call parameter type does not match function signature
In older versions of LLVM (before LLVM 15), pointers have strict types. A procedure pointer might expect a simple generic pointer like double*, but we were passing it a fixed-size array pointer like [100 x double]*. This strict type mismatch caused the backend verifier to fail.

Minimal Reproducible Example (MRE):

program test_proc_ptr
  implicit none
  abstract interface
      subroutine proc_iface(a)
          real(8), intent(inout) :: a(100)
      end subroutine
  end interface
  procedure(proc_iface), pointer :: pp
  real(8) :: x(100)
  
  call pp(x) 
end program

The Solution: To satisfy the strict LLVM verifier, I added a conditional check that inserts an explicit bitcast (CreateBitCast) right before the indirect procedure call. This bitcast converts the fixed-size array argument into the generic pointer type that the function signature actually expects. Since this is only an issue with older typed pointers, I made sure this code is only compiled for LLVM versions older than 15. For LLVM 15 and above, which use opaque pointers, this bitcast is safely ignored.

Resolving Multiple Definition Linker Errors for Compiler-Generated Helpers (PR #12174)

The Problem: When trying to link multiple compiled Fortran files together, I started seeing severe linker errors: multiple definition of `_lcompiler_...`
This happened because LFortran generates special helper functions (like _lcompiler_optimization_exponent_f64) for intrinsics. These helpers were being fully generated and exported in every single file that used them, causing the linker to complain about duplicate definitions.

Minimal Reproducible Example (MRE):

! file1.f90
module mod1
contains
  subroutine test1(x)
    real(8) :: x
    print *, exponent(x)
  end subroutine
end module

! file2.f90
module mod2
contains
  subroutine test2(x)
    real(8) :: x
    print *, exponent(x)
  end subroutine
end module

! main.f90
program main
  use mod1
  use mod2
  call test1(2.0_8)
  call test2(3.0_8)
end program

The Solution: I figured out that we could tell the linker these helper functions are identical across all files. I solved this by setting the linkage type to LinkOnceODRLinkage for any function starting with _lcompiler_. This instructs the linker to safely throw away the duplicate definitions and keep just one single copy for the final executable. This completely eliminated the multiple definition errors!
