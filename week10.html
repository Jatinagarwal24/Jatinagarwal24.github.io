GSoC 2026 @Fortran-lang

Week 10 - Namelist Pointer I/O, Preprocessor Fixes, and 100% Compilation of benchmark.f

July 18 – July 24, 2026

This week was a big milestone: I fixed two more LFortran bugs (namelist I/O of scalar pointers and a preprocessor edge case in function-like macros), and I also found and reported two genuine bugs in the benchmark.f package itself to its author, along with proposed fixes. With all of this in place, the current LFortran main branch now compiles and tests benchmark.f to 100%!

Fixing Namelist I/O of Scalar Pointer/Allocatable Variables (PR #12303 for Issue #12302)

The Problem: When a namelist group contained a scalar pointer or allocatable variable, LFortran printed the pointer itself instead of the value it points to. This produced garbage values for intrinsic types (e.g. x=4210792 instead of x=10) and even a segmentation fault for pointers to derived types with character components.

Minimal Reproducible Example (MRE):

program mre
implicit none
integer, target :: t = 10
integer, pointer :: x => null()
namelist / config / x
x => t
print *, 'plain print of x:', x
write(*, nml=config)
end program
! LFortran (before the fix)
plain print of x: 10
 &config
  x=4210792        <-- garbage: the pointer itself, not its target!
 /

! gfortran (expected)
 plain print of x:          10
&CONFIG
 X=10         ,
 /

The Solution: Since a plain print *, x printed the correct value, I knew pointer association itself was fine and the bug had to be in the namelist lowering. I traced it to build_namelist_descriptor in the LLVM backend: for scalar pointer and allocatable items, the symbol table value (which is the pointer variable's own storage) was passed directly as the data pointer in the namelist descriptor. So at runtime, the I/O routines happily read the raw address as if it were the value, and for pointers to derived types with character components they chased that bogus address and segfaulted. Interestingly, pointer/allocatable arrays didn't have this problem, because their code path already loaded through the pointer before building the descriptor. The fix was to do the same for non-character scalars: emit a load through the symtab value first, so the descriptor always points at the actual target. I also added an integration test (namelist_36) that round-trips a scalar integer pointer, a scalar real allocatable, and a pointer to a derived type with a character component (the case that previously segfaulted).

Fixing Preprocessor Failure on #define Bodies Starting Right After the Parenthesis (PR #12314 for Issue #12313)

The Problem: LFortran's preprocessor failed on function-like macros where the macro body starts immediately after the closing parenthesis, a pattern used heavily in benchmark.f's test framework. The macro was silently ignored with a #define ignored warning, and the stray line-continuation \ then crashed the tokenizer with Token '\' is not recognized.

Minimal Reproducible Example (MRE):

#define FOO(x)\
print *, x; \
print *, x;
program main
implicit none
FOO('hello')
end program

The Solution: Digging into the preprocessor, I found that the matching rules for function-like macros required at least one whitespace character between the closing parenthesis and the macro body. A definition like #define FOO(x)\ therefore matched none of the macro rules, fell through to the generic "#define ignored" branch, and the leftover \ reached the tokenizer, which has no idea what to do with a backslash. I made the whitespace optional in both the zero-argument and named-argument macro rules. While doing that, I also uncovered a second latent bug: when the old optional whitespace-plus-body group didn't match, its captures were left unset, and token(t3, t4) would read garbage, so I changed the rules to always capture the body. Finally, I hooked handle_continuation_lines into the zero-argument rule as well, so multi-line macro bodies get stitched together no matter which rule matched. I verified the expansion against gfortran -cpp and added integration test cpp_pre_18 covering both continuation lines and bodies starting immediately after ).

Finding Real Bugs in benchmark.f Itself

Not every failure this week was LFortran's fault! While debugging the test suite, I discovered two genuine bugs in the benchmark.f package and reported them upstream to the author with minimal reproducers and proposed patches. Interestingly, LFortran's stricter runtime checks are what exposed both of them, while gfortran silently masks them.

1. Referencing an Unallocated Allocatable String (benchmark.f Issue #7)

The Problem: In src/Argument.f90, arg_new_from_string calls len_trim(display%chars) without checking whether chars is allocated. But parse_names in src/Benchmark.f90 returns a fixed-size names(7) array where only the parsed slots get chars allocated, and the run methods construct arguments for all 7 slots regardless. Passing an unallocated allocatable to len_trim is non-conforming (F2018 15.5.2.12). gfortran silently returns 0 so it goes unnoticed, but LFortran's runtime check correctly stops with Argument 1 of len_trim ... is unallocated, breaking test_calling.

Minimal Reproducible Example (MRE):

module string_m
    implicit none
    type :: string
        character(:), allocatable :: chars
    end type
contains
    ! mimics parse_names: fixed-size result, only parsed slots get assigned
    function parse_names(n) result(names)
        integer, intent(in) :: n
        type(string) :: names(n)
        names(1)%chars = 'a1'
        names(2)%chars = 'a2'
    end function

    ! mimics arg_new_from_string (src/Argument.f90)
    subroutine arg_new_from_string(display)
        class(string), intent(in) :: display
        print *, 'len_trim =', len_trim(display%chars)
    end subroutine
end module

program mre
    use string_m
    implicit none
    type(string) :: names(7)
    names = parse_names(7)
    call arg_new_from_string(names(1))   ! ok
    call arg_new_from_string(names(4))   ! never assigned -> non-conforming
end program

The Fix: I proposed a patch guarding every access with allocated() checks, both in arg_new_from_string and in the same unguarded pattern in src/String.f90 (string_assign_character and string_len, which now returns 0 for an unallocated string). The author confirmed the bug and merged the fix.

2. Passing class(*) Arguments Through an Implicit-Interface Procedure Pointer (benchmark.f Issue #8)

The Problem: In src/Method.f90, invoke_a0 calls the benchmarked procedure as call this%f(arg1, arg2), where f is a procedure(), nopass, pointer (implicit interface) and arg1/arg2 are class(*) associate-names. Polymorphic actual arguments require an explicit interface (F2018 15.4.2.2). With no interface, there is no defined way to pass the value, so the callee receives an address instead of the data. I verified by printing the arguments inside the benchmarked routine: instead of a=1.0d-6, m=30 it received a ≈ 5e-310 and m in the millions, under both gfortran and LFortran! So the benchmark was actually timing a garbage-bounded loop, and nothing caught it since outputs are never validated.

Minimal Reproducible Example (MRE):

module method_m
    implicit none
    type :: arg
        class(*), allocatable :: value
    end type
    type :: method
        procedure(), nopass, pointer :: f => null()
        type(arg), allocatable :: args(:)
    contains
        procedure :: invoke
    end type
contains
    subroutine work(a, n)
        real(8), intent(in) :: a
        integer, intent(in) :: n
        if (a == 1.0d-6 .and. n == 30) then
            print *, 'OK: work received a=', a, ' n=', n
        else
            print *, 'FAILED: work received a=', a, ' n=', n
            stop 1
        end if
    end subroutine

    type(method) function make(f, a1, a2) result(mtd)
        procedure() :: f
        class(*), intent(in) :: a1, a2
        mtd%f => f
        allocate(mtd%args(2))
        allocate(mtd%args(1)%value, source=a1)
        allocate(mtd%args(2)%value, source=a2)
    end function

    ! mimics Method.f90 invoke_a0 case (2)
    subroutine invoke(this)
        class(method), intent(inout) :: this
        associate(arg1 => this%args(1)%value, arg2 => this%args(2)%value)
            call this%f(arg1, arg2)
        end associate
    end subroutine
end module

program mre
    use method_m
    implicit none
    type(method) :: mtd
    mtd = make(work, 1.0d-6, 30)
    call mtd%invoke()
end program
! LFortran
FAILED: work received a= 2.0820123942008466E-317  n= 4214080

! gfortran (same garbage!)
 FAILED: work received a=   5.2277794488084471E-310  n=   619719024

The Fix: Along with the report, I proposed a patch to the test suite (accepting LFortran's output formatting alongside gfortran's and making a platform-dependent file-size check tolerant). We also had a nice discussion about subroutine-vs-function procedure arguments across compilers: gfortran rejects passing a function where a subroutine is expected, while LFortran is more permissive through implicit interfaces. The author fixed the bugs and pointed out that the safe way to use the library is the explicit-interface set_caller mechanism, which is exactly the code path I fixed in LFortran back in Week 9!
