Week 5 - Fixing Procedure Pointer Regressions

June 13 – June 19, 2026

This week was heavily focused on pushing past two major compilation milestones in benchmark.f. I tackled both the 86% and 91% blockers, which required deep dives into the compiler's LLVM code generation and ASR (Abstract Semantic Representation) type synchronization logic. Here is a breakdown of the two major Pull Requests that resolved these issues: PR #11806 and PR #11874.

PR #11806: Resolving LLVM IR Verification Failures

At the 86% compilation mark, the compiler was crashing with LLVM IR verification failures. PR #11806 tackled two distinct issues that were causing these crashes:

  • Intrinsic Type Mismatches: We encountered an issue where the date_and_time intrinsic was generating invalid LLVM IR. The fix involved correcting descriptor-based array type initialization and ensuring that variable kinds were properly cast (e.g., adding strict cast_kind rules) before being passed to the backend.
  • Procedure Pointer Signatures: The second part of the fix addressed a compiler crash that happened when passing a specific function to a generic procedure argument (like procedure() :: f). The backend strictly rejected the call because the signature of the function we were passing didn't exactly match the generic, empty signature expected by the procedure argument.

    Initially, I tried to force these types to match by simply casting (bitcasting) them directly during the LLVM code generation phase. However, after discussions with Saurabh and Ondřej, we realized this wasn't the correct approach. The compiler's abstract representation (ASR) needed to understand and validate these types first, rather than just forcing them to fit at the very end.

    To find the best path forward, we analyzed how other major compilers like GFortran and Flang handle this kind of runtime polymorphism. Flang uses modern "opaque" pointers, while GFortran performs a cast at the call site. Since LFortran uses explicitly typed pointers, a cast at the call site was unavoidable, but we needed to make it safe and predictable.

    Instead of blindly casting whenever types didn't match, I updated the compiler to check the ASR first. If it sees that a parameter is specifically designed to be a generic procedure interface, it now safely casts the incoming function to a generic pointer type (like void ()*) right at the call site. This perfectly aligned our implementation with standard compiler behaviors while ensuring the compiler strictly understands what it's doing!

PR #11874: The 91% Blocker & Implicit Interface Sync

With the 86% blocker resolved, compilation progressed to 91% before hitting a severe ASR verify pass error: 

ASR verify pass error: Number of argument types in FunctionType must be exactly same as number of arguments in the function

This was tracked in Issue #11873. The root cause was that when compiling procedure pointers acting as dummy arguments with implicit interfaces (e.g., procedure(), nopass, pointer :: caller => null()), the ASR verifier and LLVM backend were failing to keep the function signatures synchronized with the actual arguments.

To fix this, I submitted PR #11874, which required several coordinated changes during the compiler's semantic analysis phase.

First, we noticed that different parts of the code were creating separate copies of the same function signature in memory. When one part updated the signature, the others didn't see the change. I modified the code to strictly reuse the exact same memory object for these interfaces. This stopped the "type divergence," ensuring that any update was instantly seen by all variables sharing that interface.

Next, to satisfy the compiler's internal verifier, I added logic to explicitly create internal dummy variables for these interfaces so that the number of arguments perfectly matched the expected list of argument types.

The most crucial architectural fix was adding a new synchronization sweep at the very end of the semantic analysis. This loop specifically targets auto-generated interfaces and synchronizes the types of their internal dummy variables with their fully resolved argument types. To ensure we didn't accidentally break standard user-defined interfaces, I added a strict filter based on internal naming conventions.

Finally, I made sure that whenever a procedure variable's arguments are successfully resolved inside a function call, that updated type information is actively propagated back up to the parent function. This keeps everything perfectly in sync from top to bottom!

Progress & Next Steps

We've made massive strides this week in overcoming these two compilation hurdles, but there is still some remaining work to finalize them:

  • For PR #11806: Only a few minor things remain. Specifically, we need to add #ifdef guards to correctly query the LLVM version and ensure compatibility across different LLVM releases.
  • For PR #11874: I will be iterating on the current fix and refining the synchronization approach with Assem in the coming days.

Once these final touches are completely wrapped up, we will finally move forward past the 91% compilation mark and see what the next error in benchmark.f holds for us!

Time Log: I worked around 33 hours this week, diving deep into runtime polymorphism across different compilers, implementing ASR-driven type checking, and resolving complex LLVM signature mismatches for procedure pointers.
