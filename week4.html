Week 4 - Resolving procedure pointer MREs and new LLVM verification errors

June 6 – June 12, 2026

This week was another intense debugging week, focusing on moving past the procedure pointer codegen crash from last week and tackling new LLVM verification errors. We also uncovered a bug introduced by recent upstream commits in benchmark.f. Here's a breakdown of the week's work.

Refining the Procedure Pointer Codegen Fix (PR #11728)

The update: Last week, I started working on a fix for a procedure pointer codegen crash in PR #11728. During the review process, we realized that my previous Minimal Reproducible Example (MRE) was slightly incorrect and wasn't fully capturing the essence of the bug as it occurred in benchmark.f.

Thankfully, Saurabh Kumar stepped in and helped provide the correct MRE. It turned out that the crash was happening during a subroutine call through a procedure pointer in a derived type, while the caller was null(). I updated the tests to use this correct MRE, and the PR was approved.

The 53% Blocker: Generic Interface and Structure Constructor Name Clash (Issue #11793)

The problem: After getting the previous fixes in, we ran into a new issue that halted compilation at 53%. This bug was introduced by recent commits from the author of benchmark.f. The compiler was hitting a segmentation fault when dealing with a derived type that shares a name with a generic interface (Issue #11793). This is perfectly valid Fortran, often used for extending types or creating custom constructors.

The root cause: Inside LFortran, when resolving a generic procedure call in create_GenericProcedureWithASTNode, if the resolution failed (i.e., no matching specific procedure was found), the compiler would incorrectly fall through to checking if it was an intrinsic function. It completely missed the fallback step: checking if the call was actually a structure constructor for a derived type of the same name. This led to a null pointer being passed down and causing a segfault in create_FunctionCall.

The fix: In PR #11794, I added the missing structure constructor fallback logic. Now, if generic procedure resolution fails, LFortran checks for a derived type with that name and correctly builds a DerivedTypeConstructor instead of crashing. With this fix, we were immediately back to 86% compilation!

The 86% Blocker: LLVM Verification Errors (Issue #11805)

The problem: Back at 86%, we hit the LLVM verification errors I mentioned at the end of last week (Issue #11805). Specifically, two distinct LLVM IR verification failures were stopping us:

  1. date_and_time intrinsic type mismatch: When providing a 64-bit integer array (integer(8)) to the values argument of date_and_time, LFortran generated LLVM IR that tried to directly store 32-bit runtime values into the 64-bit array, resulting in a Stored value type does not match pointer operand type! error.
  2. Procedure pointer signature mismatch: When passing a procedure to a struct constructor containing an implicit-interface procedure pointer field, the generic resolution could select the wrong interface or generate a mismatched LLVM function type signature, causing a Call parameter type does not match function signature! error.

The fix: I tackled both issues in PR #11806:

  • For the date_and_time bug, I updated instantiate_DateAndTime in intrinsic_subroutines.h to explicitly cast the runtime i32 values to the target array's element type before populating the ArrayConstant. This ensures the type semantics match correctly before codegen.
  • For the procedure pointer issue, my initial understanding was to resolve the signature mismatch by adding an explicit LLVM BitCast during codegen when assigning to procedure pointers. However, during the review, Saurabh pointed out that the type semantics should correctly match through the ASR instead of relying on an LLVM-level cast. Following his guidance, I fixed it properly at the ASR level by tightening the argument_types_match logic in asr_utils.cpp to strictly require FunctionType_t for both formal and actual arguments during implicit interface matching.

This PR is currently under review.

The 91% Blocker: ASR Verify Pass Error

Once PR #11806 is merged into main, we will successfully move past the 86% mark and hit 91% compilation of benchmark.f. 

ASR verify pass error: Number of argument types in FunctionType must be exactly same as number of arguments in the function

This ASR verify pass error points to a new issue where the FunctionType signature doesn't match the actual arguments passed to the function, specifically triggered around an implicit-interface procedure pointer (caller => null()). Resolving this verification failure will be my next primary target as we push toward compiling the entire package!

Time Log: I worked around 40 hours this week, including significant time on debugging deep LLVM codegen issues and iterating on fixes with reviewers. Also worked on random bugs reported by other users.
