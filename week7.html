Week 7 - Resolving Internal Compiler Errors and Deserialization Issues

June 27 – July 3, 2026

This week, I focused heavily on resolving complex Internal Compiler Errors (ICEs) that popped up while compiling benchmark.f, specifically dealing with ASR symbol table scopes and module deserialization bugs related to procedure pointers and intrinsic wrappers.

Fixing ASR Symbol Table and Deserialization Bugs

In my continuous effort to achieve full compilation of benchmark.f, I discovered and resolved two tricky bugs regarding how LFortran handles scopes for intrinsic function wrappers and module serialization.

1. ASR Verify Fails for Intrinsic Wrappers in Nested Blocks (Issue #12043 / PR #12044)

The Problem: I encountered an Internal Compiler Error when calling `get_environment_variable` with a string slicing expression from within a nested scope, like a block construct. The ASR verifier threw a scoping error: Var::m_v 'ilen' cannot point outside of its symbol table This happened because the compiler generates a synthetic wrapper subroutine for the intrinsic function and hoists it to the global scope. However, it mistakenly captured the explicit length dimension containing the local variable ilen, causing an out-of-scope reference!

Minimal Reproducible Example (MRE):

program p
  implicit none
  integer :: ilen
  character(10) :: ctmp
  character(10) :: str = "hello"
  ilen = 5
  block
    call get_environment_variable(str(:ilen), ctmp)
  end block
end program p

The Solution: To fix this scoping bug, I updated the synthetic wrapper generation logic. Specifically, in intrinsic_subroutines.h for instantiate_GetEnvironmentVariable, I forced the string arguments (name and value) passed to the wrapper to use an AssumedLength character type. By doing this, the wrapper no longer captures the local variable defining the explicit string length. I isolated this change to avoid altering other arguments (like explicit-shape arrays), ensuring LLVM codegen signatures remained perfectly intact.

2. Deserialization Failure for Implicit Interface Procedure Pointers (Issue #12045 / PR #12046)

The Problem: Another severe ICE occurred during module deserialization. When importing a module containing a derived type with a procedure pointer (which lacked an explicit interface) into another module, and then utilizing it in the main program, the compiler crashed with: Deserialization failed: symbol '__caller_iface_implicit' references symbol table with ID 2 which has not been deserialized yet.

Minimal Reproducible Example (MRE):

module mod_a
  implicit none
  type :: type_a
    procedure(), nopass, pointer :: caller => null()
  end type
end module

module mod_b
  use mod_a
  implicit none
  type :: type_b
    procedure(), nopass, pointer :: caller => null()
  end type
end module

program p
  use mod_b
  implicit none
  type(type_b) :: x
end program

The Solution: The crash was caused by the synthetic __caller_iface_implicit interface (generated by the compiler to represent the implicit interface) leaking into the module's public Application Binary Interface (ABI). During deserialization, this caused an out-of-order symbol table reference because the compiler tried to resolve internal metadata that shouldn't have been exported. I resolved this by correctly marking the synthetic interface as Private (ASR::accessType::Private) in ast_common_visitor.h. Now, the compiler properly treats it purely as internal ASR type-checking metadata, preventing the critical deserialization failure!

Progress & Next Steps

With these complex symbol table and serialization bugs out of the way, LFortran is becoming significantly more robust in handling nested scopes and complex module dependencies. Most importantly, once these two PRs get merged into the main branch, we will successfully be compiling 92% of the benchmark.f package! My next steps involve continuing the push to reach 100% compilation and running the complete test suite to uncover any further edge cases.
