Compiling Benchmark.f and feq-parse with LFortran

GSoC 2026 @Fortran-lang

by Jatin Agarwal

Week 9 - Fixing Procedure Pointers, Allocatable Scalars, and Completing feq-parse

July 11 – July 17, 2026

This week involved a mix of fixing LLVM codegen for procedure pointers, refining my previous approach with ArrayPhysicalCast, and resolving a segmentation fault that allowed me to finally compile the feq-parse package!

Fixing Procedure Pointer Type Mismatch in Cross-Struct Association (PR #12184 for Issue #12182)

The Problem: When associating a typed procedure pointer from one struct member to a generic procedure() pointer slot in another struct (e.g. mtd%caller => this%caller), the LLVM codegen stored the value without bitcasting. This caused a Stored value type does not match pointer operand type verification failure on LLVM < 15 which still uses typed pointers.

Minimal Reproducible Example (MRE):

module mod_method
  implicit none
  type :: method
      procedure(), nopass, pointer :: caller => null()
  end type
end module

module mod_runner
  use mod_method
  implicit none
  type :: runner
      procedure(), nopass, pointer :: caller => null()
  contains
      procedure, pass(this) :: set_caller
      procedure, pass(this) :: benchmark_a1
  end type
contains
  subroutine set_caller(this, caller)
      class(runner), intent(inout) :: this
      procedure() :: caller
      this%caller => caller
  end subroutine

  subroutine benchmark_a1(this, a1, f)
      class(runner), intent(inout) :: this
      class(*), intent(in) :: a1
      procedure() :: f
      type(method) :: mtd
      if (associated(this%caller)) mtd%caller => this%caller
  end subroutine
end module

The Solution: The root of the problem was that the compiler's Abstract Semantic Representation (ASR) represented procedure pointer members as Pointer(FunctionType). During the translation to LLVM IR, this fell into a generic fallback branch inside visit_Associate. To fix it, I explicitly cast the concrete function pointer to a generic void()* slot using an LLVM bitcast. I also added a version guard (#if LLVM_VERSION_MAJOR < 15) so this cast only applies to older LLVM versions that still enforce typed pointers, satisfying the LLVM verifier.

Iterating on ArrayPhysicalCast Implementation (PR #12110)

The Problem: Last week, I worked on fixing a procedure pointer verification failure by inserting an explicit bitcast at the LLVM IR level. However, after discussing with Assem, I realized that the root cause was actually earlier in the pipeline: the ArrayPhysicalCast nodes were being prematurely removed during ASR processing.

The Solution: Instead of using a quick bitcast hack in the LLVM backend like I tried initially, I went back to the ASR level. By stepping through make_ArrayPhysicalCast_t_util, I found exactly where the compiler was mistakenly stripping away ArrayPhysicalCast nodes. I modified the logic to preserve these casts whenever they're needed for procedure pointer interfaces. This way, the fix is much cleaner and happens right in the frontend where it belongs.

Once this PR and the previous ones are merged into the main branch, I will be fully compiling the benchmark.f package and moving on to evaluating the test suite!

Fixing Segmentation Fault in Allocatable Scalar Assignment (Issue #12220)

The Problem: I encountered a severe issue where LFortran would throw a Segmentation fault (core dumped) when trying to assign between allocatable scalar variables (e.g. b = a where both are integer, allocatable). This was one of the final blockers for compiling the feq-parse package.

Minimal Reproducible Example (MRE):

program mre_alloc_int
  implicit none
  integer, allocatable :: a, b
  a = 10
  b = a
  print *, b
end program mre_alloc_int

The Solution: I dug into the LLVM code generation in asr_to_llvm.cpp and traced the crash down to how assignments were lowered. The compiler was blindly dereferencing the destination's pointer to store the new value. But for allocatable scalars, if the variable hasn't been allocated yet, its pointer is null! Dereferencing a null pointer is a classic recipe for a segfault. To fix this, I added a call to check_and_allocate_scalar() before the actual copy happens for basic intrinsic types (Integer, Real, Complex, Logical). This ensures memory is safely dynamically allocated on the fly if it doesn't exist yet, cleanly resolving the crash.
