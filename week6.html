Tackling New Implicit Interface Bugs

As I pushed forward, I uncovered two new compiler bugs, both related to how LFortran handles procedure pointers and implicit interfaces.

1. ASR Verify Pass Error with StructType (Issue #11974 / PR #11975)

The Problem: When creating an implicit interface for a procedure pointer, if one of the dummy arguments happens to be a derived type (class or type), the ASR verifier would throw an error:
Variable __caller_iface_implicit_arg_1 of type StructType must have a type declaration.

Minimal Reproducible Example (MRE):

module m1
  implicit none
  type :: runner
    procedure(), nopass, pointer :: caller => null()
  contains
    procedure, pass(this) :: set_caller
  end type
contains
  subroutine set_caller(this, caller)
    class(runner), intent(inout) :: this
    procedure() :: caller
    this%caller => caller
  end subroutine
end module

program p
  use m1
  implicit none
  type(runner) :: br
  call br%set_caller(upper_caller)
contains
  subroutine upper_caller(a)
    class(*), intent(in) :: a
  end subroutine
end program

The Solution: The issue was that the implicitly generated procedure interface was missing the critical m_type_declaration linkages for its StructType dummy arguments. To fix this, I updated the AST-to-ASR conversion phase to properly construct these linkages. When synthesizing dummy arguments, the compiler now checks if an argument is of a StructType. If it is a standard derived type, it searches the parent scope to find and link the existing structure definition. For unlimited polymorphic types (like class(*)), it specifically resolves or dynamically generates a special ~unlimited_polymorphic_type symbol to act as the declaration. Supplying this explicit type declaration to the newly created dummy variable completely eliminated the verification failures.

2. ICE with Imported Implicit Interfaces (Issue #11976 / PR #11979)

The Problem: I discovered an Internal Compiler Error (ICE) that triggered an AssertFailed crash. This happened when a procedure with an implicit interface was defined inside a module, imported into the main program via a use statement, and then used alongside a local type containing a procedure pointer.

Minimal Reproducible Example (MRE):

module module_a
  contains
  subroutine sub_a(caller)
    procedure() :: caller
  end subroutine
end module

program main
  use module_a
  type :: runner
      procedure(), nopass, pointer :: caller => null()
  end type
  type(runner) :: r
end program

The Solution: The root cause of the crash was a failing type cast (ASR::down_cast) during semantic analysis. When extracting the function signature to synchronize implicit interfaces, the compiler blindly downcasted the interface symbol to a Function_t. However, because the symbol was imported via a use statement, it was actually wrapped in an ExternalSymbol. Downcasting an external symbol directly to a function triggered the assertion failure. I fixed this by adding a crucial step to unwrap the symbol using ASRUtils::symbol_get_past_external(). This correctly resolves the imported symbol to its underlying function before the cast, allowing the compiler to safely extract the function signature without crashing!
