GSoC 2026 @Fortran-lang

by Jatin Agarwal

Week 2 - Fixing generic procedure dispatch and polymorphic allocation ICEs

May 24 – May 29, 2026

This week was all about moving past the serialization bugs from Week 1 and hitting the next set of problems in benchmark.f. Last week we ended at a semantic error: "Arguments do not match for any generic procedure, run". This week I dug into that, fixed it, and then immediately ran into another ICE — this time deep inside the LLVM codegen for struct finalization. Here's how it went.

Issue #1: Generic procedure not matching implicit interface arguments

The problem: In benchmark.f, there's a type-bound generic procedure called run. One of its specific implementations takes a procedure() dummy argument (an implicit interface — basically saying "I accept any procedure, I don't care about its signature"). When you call br%run(100000000000_8, compute_pi), LFortran throws a semantic error saying it can't match the generic. But gfortran compiles it fine.

I extracted a minimal reproducer (Issue #11624) and started digging into the matching logic.

Finding the root cause: The generic dispatch logic lives in argument_types_match in asr_utils.cpp. When matching arguments, it compares the type of the actual argument (what you pass) against the type of the formal argument (what the subroutine expects). The problem was that it was doing a strict type equality check even when the formal argument is procedure() — which by Fortran standard should accept any procedure. A procedure() dummy has zero argument types and Interface deftype, but the matching code didn't know that means "accept anything".

The fix: I added checks in both the Variable_t branch and the Function_t branch of argument_types_match. If the formal argument's type declaration is an implicit interface (identified by _iface_implicit in the declaration name), or if the actual argument is an implicit interface procedure (FunctionType with 0 arg types and Interface deftype), skip the strict type matching. The actual compatibility is checked later at the call site anyway.

But then it got complicated. The semantic error was gone, but now LFortran was crashing during LLVM codegen. The issue was in asr_to_llvm.cpp — when the compiler tries to generate code for the call, it needs to look up the function pointer for compute_pi. There's a function called instantiate_function that pre-declares Interface functions, but it had a skip condition: if the function has class(*) arguments and m_module is false, skip it. This was originally meant for submodules only, but the check was too broad — it also skipped functions inside programs.

After Saurabh Kumar pointed this out during the review, I fixed it by adding a parent scope check — only skip when the parent is NOT a Program_t. This way program-scoped interface functions with class(*) args get correctly pre-declared.

I also had to fix the function lookup logic for Interface functions in codegen. Interface functions can appear in two contexts: as procedure dummy arguments (stored in llvm_symtab_fn_arg) or as standalone interface declarations (stored in llvm_symtab_fn). The code needed to check llvm_symtab_fn_arg first and fall back to llvm_symtab_fn. During review, we tested reverting this to the original code and found the original three-branch structure was already logically equivalent — so we kept the original.

This was merged as PR #11625.

Issue #2: ICE in allocate with polymorphic source and finalizer

The problem: With the generic dispatch fixed, benchmark.f moved further but immediately hit another crash: 

AssertFailed: type_to_check_against == llvm_SSA->getType()

This was happening inside call_struct_finalize_fn during an allocate(x%p, source=y) where x%p is a class(t), pointer member and the type t has a finalizer. I extracted a minimal reproducer (Issue #11675) and started tracing.

The debugging process: This one was tricky. The crash was in the LLVM codegen, deep inside the struct deep copy logic. When you do allocate(x%p, source=y), the compiler needs to:

  1. Call the finalizer on the destination (to clean up old data)
  2. Deep copy the source into the destination

The finalizer call was failing because it expected a plain struct pointer but was getting a class wrapper pointer (which has an extra vtable pointer field). I first tried to fix call_struct_finalize_fn by detecting if the pointer is a class wrapper and extracting the inner struct. But that was fragile.

The real insight: While tracing the code, I noticed something weird about the deepcopy function signature. It takes parameters named asr_dest_type (4th param) and asr_src_type (5th param). But the callers were consistently passing the source type as the 4th argument and dest type as the 5th! The parameter names in the function signature are swapped relative to what callers actually pass.

This meant that inside struct_deepcopy, the is_dest_class flag was actually reflecting the source's type, and is_src_class was reflecting the destination's type. For cases where both sides are class types (like the existing pointer_04 test), this didn't matter — the swap cancelled out. But for our case where source is class(t) and dest is a plain type(t) pointer, the flags were wrong, causing the code to try extracting a class wrapper from a plain struct pointer.

The fix (two parts):

  • In struct_deepcopy: Swapped which ASR flag controls which extraction — is_dest_class now controls source extraction (because it actually reflects the source), and is_src_class controls dest extraction. Also used the LLVM getClassType and get_type_from_ttype_t_util to verify types before doing GEP operations.
  • In call_struct_finalize_fn: Replaced the strict type assertion with a bitcast to match the finalizer function's expected parameter type. This handles both class wrapper and plain struct pointers correctly.

This was opened as PR #11676.

Progress and Next Steps

The compilation of benchmark.f is still at 86% — same as last week. But that doesn't mean nothing happened. Two major blockers that were stopping us at 86% have been addressed this week. The generic dispatch fix is merged, and the struct finalizer ICE fix is up for review. However, as soon as we got past those, a third blocker showed up — an argument count mismatch in convert_call_args. Next week, I'll be tackling that one and hopefully we will move forward to compile it further.

Time Log: I worked around 28 hours this week, including significant time on back-and-forth with reviewers to refine the approach and ensure no regressions. I also worked on fixing some other bugs reported by users.
