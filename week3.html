Week 3 - Fixing procedure pointer codegen and polymorphic finalizer ICEs

May 30 – June 5, 2026

This week was packed. We started by tackling the struct finalizer ICE from last week (Issue #11675), got it through code review and merged, then immediately moved to a new blocker, a procedure pointer codegen crash (Issue #11727). Here's how it all went.

Issue #1: Getting the struct finalizer ICE fix merged (Issue #11675, PR #11676)

Recap: Last week I opened PR #11676 to fix the ICE in call_struct_finalize_fn during allocate(x%p, source=y) when x%p is a polymorphic class(t), pointer member and the type t has a finalizer. This week was about addressing code review feedback from Assem Medhat to get this merged.

Assem's review (3 key points):

  • Root cause is wrong ASR, not just LLVM: Assem pointed out that the real root cause was the m_is_cstruct flag being set incorrectly in the ASR. For the Fortran code: 
    type(t) :: x
class(t), pointer :: xxx
allocate(xxx, source=y)  ! This works
allocate(x%p, source=y)  ! This doesn't work
    x%p had its m_is_cstruct set to true, while xxx (which is equivalent) correctly had it set to false. I fixed this by changing it to false, aligning the behavior with xxx.
  • Remove the bitcast in call_struct_finalize_fn: My initial fix used a bitcast to convert the pointer to match the finalizer's expected type. Assem said: "ptr should match function's signature. We shouldn't bitcast. LLVM gen. would warn us if we use totally wrong argument, with bitcast we'd lose that." So I removed the bitcast and instead correctly used llvm_utils->create_gep2(class_llvm_type, ptr, 1) with CreateLoad to properly extract the data payload from the class wrapper, maintaining exact type safety.
  • Fix the memory leak with finalize_before_deallocate: Assem identified that when the target struct has allocated-allocatable members (e.g., integer, allocatable :: val), only finalizing the outer layer would leak the inner allocatables. He suggested using finalize_before_deallocate instead. I deleted the manual if(m_is_cstruct) block with lfortran_free(inner_ptr) and restored the standard finalization flow, which handles nested allocatables correctly.

After addressing all three review comments, this PR got merged and with this done, benchmark.f moved past the finalizer crash and hit the next blocker.

Issue #2: Procedure pointer codegen crash (Issue #11727)

The problem: With the finalizer fix merged, benchmark.f hit the next ICE, an assertion failure during LLVM codegen: AssertFailed: arg_idx < func_subrout->n_args. This was happening when calling a procedure(), nopass, pointer struct member with actual arguments.

I extracted a minimal reproducer: 

program simple
    implicit none
    type :: runner
        procedure(), nopass, pointer :: caller => null()
    end type
    type(runner) :: br
    call br%caller(100)
end program

Root cause: When you declare a procedure pointer as procedure() (with empty parentheses), you're saying "this can point to any procedure, I don't care about its signature." The Fortran standard calls this an implicit interface. LFortran's ASR records this as a Function_t with zero formal arguments (n_args = 0).

But at the call site call br%caller(100), you're passing one actual argument. During LLVM codegen, the function convert_call_args loops over the actual arguments and tries to look up the corresponding formal argument for each one. It asserts arg_idx < func_subrout->n_args, which fails since 1 < 0 is false.

The fix (four parts in asr_to_llvm.cpp):

  • Guarded set_func_subrout_params: Added a bounds check if (i < func->n_args) before calling the parameter-resolution function. When the declared function has fewer args than the call provides, skip the lookup.
  • Short-circuited implicit-interface args: When orig_arg is nullptr (because the procedure has no declared params), we evaluate the argument expression, wrap it in an alloca for pass-by-reference, push it to the args list, and continue skipping all the complex parameter-matching logic that assumes a non-null orig_arg.
  • Guarded the second orig_arg lookup: There was a second assertion arg_idx < func->n_args deeper in the function. Added the same bounds check here and allowed the Variable symbol type to fall through without asserting false.
  • Reconstructed LLVM FunctionType: The declared function type has 0 parameters (void ()*), but the actual call passes arguments. I built a new LLVM FunctionType from the actual argument types and bitcasted the callee pointer to match. This is the standard approach because LLVM requires strict type matching for calls, and bitcasting is how compilers bridge the gap between Fortran's flexible implicit interfaces and LLVM's strict type system.

This was opened as PR #11728 and is currently under review.

What's the new error?

After fixing the procedure pointer codegen crash, benchmark.f now gets further but reveals a new set of pre-existing LLVM type mismatch errors: 

Call parameter type does not match function signature!
void (%"~unlimited_polymorphic_type"*)* @compute_pi
 void ()*

These errors were always there, they were just hidden because the compiler crashed (ICE) before reaching LLVM verification. Now that our fix prevents the crash, the compilation proceeds further and reveals these deeper issues. The problem is that when a typed procedure (like compute_pi which takes class(*)) is passed to a procedure() dummy parameter, LFortran generates void ()* for the dummy but the actual procedure has a typed signature void (%"~unlimited_polymorphic_type"*)*, causing a type mismatch.

This will be the focus of next week's work: fixing the LLVM type generation for procedure() dummy arguments so they correctly match the actual procedure signatures.

Progress and Next Steps

The compilation of benchmark.f remains at 86%, the same number as the last two weeks. But significant progress has been made underneath: we've cleared multiple major ICEs that were blocking at 86%, and the new error (LLVM type mismatch for procedure arguments) is the first one that isn't a crash, it's a type system correctness issue. Next week, I'll be extracting a minimal reproducer for this new error and working on the fix.
